### 📝 通用技术科普写作 Prompt 模板

请写一篇面向[搞明白这个问题之前的我自己以及其他可能有这个困惑的人]（不要在文中刻意强调这个）的技术博客，**主题为：**

> **“clash-for-linux 是怎么提供代理的？”**  

#### 🎯 写作目标
帮助读者从熟悉的使用经验出发，**浅浅地往下走一步**，理解造成差异的最直接原因，**不深入底层实现细节**，但能建立基本因果认知。

#### ✍️ 风格与语气
- 语言**朴素、平实、克制**，避免刻意活泼、拟人化、夸张或网络流行语；  
- 可在结尾或关键处加入**一丝轻微、自然的小幽默**，但绝不喧宾夺主；  
- 技术术语**仅在必要时引入**，并立即用日常语言解释；  
- 重点不是展示知识，而是**消除困惑**。

#### ⚠️ 关键原则
- **只走一步**：比如解释到“服务器是否支持”“文件是否已存在”这一层，不继续追问“那服务器怎么支持的？”  
- **用户视角优先**：始终从“用户体验到什么”出发，而非从系统架构讲起；  
- **有用 > 完备**：宁可少说，也不堆砌无关细节；  
- **让读者觉得“哦，原来是这样”，而不是“我得再查查”。**



----

#### 🧱 结构建议（按认知逻辑展开）

1. **引言：从两个对比的真实场景切入**
   描述用户常遇到的两种不同体验（一个顺利，一个受挫），引出核心问题。
2. **用户能观察到的线索**
   列出 1–2 个普通用户可感知的外在线索（如界面提示、行为差异），作为判断依据。
3. **背后最直接的原因**
   解释这些差异背后的**第一层技术原因**（通常与服务端/系统状态有关），**止步于此，不继续拆解协议、算法或源码**。
4. **如何提前判断或应对**
   提供一个简单、无需安装工具的操作方法（如用浏览器 DevTools、看 URL 特征等）。
5. **实用建议**
   给出几条具体、可操作的建议，帮助用户规避问题或做出更好选择。
6. **结语：呼应开头，带一点余味**
   用一句简洁的话总结本质，可略带哲思或轻幽默。
